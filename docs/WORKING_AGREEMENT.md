# Рабочие правила проекта

Актуальный handoff для нового чата: `docs/PROJECT_HANDOFF_2026-06-23.md`. Его нужно читать первым, чтобы не восстанавливать контекст по переписке.

Этот файл фиксирует правила работы по проекту. Если правила не были явно изменены в этом файле, они считаются действующими по умолчанию.

## 1. Правила ведения проекта

1. Все важные решения фиксируются в файлах проекта, а не только в переписке.
2. После каждого значимого шага обновляются проектные документы:
   - что сделано;
   - что отложено;
   - какие новые идеи или хотелки появились;
   - какие решения были приняты и почему.
3. При разработке нужно регулярно делать локальные git-коммиты по логическим этапам, а не откладывать всё в один большой коммит.
4. Если удаленный репозиторий еще не создан или не подключен, это должно быть отдельно зафиксировано в журнале.

## 2. Инженерные требования по умолчанию

1. Любой прикладной код должен сопровождаться тестами.
2. Для ключевых пользовательских и системных сценариев должно быть предусмотрено логирование.
3. Логирование обязательно для:
   - ошибок;
   - внешних интеграций;
   - AI-вызовов;
   - платежных сценариев;
   - ключевых действий в продуктовой воронке.
4. Нельзя считать задачу завершенной, если код невозможно адекватно проверить.
5. Если тесты пока не написаны, это должно быть явно отмечено как незавершенная часть, а не потеряно.

## 3. Что обязательно фиксировать в документации

1. Что реализовано.
2. Что сознательно отложено.
3. Какие новые пожелания появились по ходу работы.
4. Какие риски, допущения и ограничения актуальны сейчас.
5. Какие следующие шаги логичны после текущего состояния.

## 4. Git и публикация

1. Работа ведется в git-репозитории.
2. Коммиты должны быть понятными и отражать логический шаг работы.
3. Нужно стремиться к настройке push в удаленный репозиторий.
4. Если для push требуется помощь пользователя, нужно явно указать, что именно нужно сделать:
   - создать репозиторий;
   - дать URL репозитория;
   - подтвердить способ авторизации.

## 5. Как интерпретировать эти правила дальше

Эти правила считаются постоянной рабочей договоренностью для проекта, чтобы их не нужно было повторять в каждом новом сообщении.

## 6. Формат объяснений

1. Объяснения даются с учетом того, что пользователь может быть новичком.
2. Если есть риск спутать команды разных систем, нужно явно говорить, для какой оболочки команда предназначена.
3. Для операций с Git, терминалом, ключами, окружением и инфраструктурой нужно давать пошаговые инструкции без лишних сокращений.
4. Если действие требует прав администратора, это нужно писать заранее.
