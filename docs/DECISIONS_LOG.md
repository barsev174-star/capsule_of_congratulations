# Журнал решений

## 2026-07-20

### Решение: scroll-анимации демонстрационной страницы — собственный модуль на IntersectionObserver, без библиотек

Принято:

1. Для reveal-анимаций `/example` создан переиспользуемый модуль `src/components/scroll-reveal/` (хук `useScrollReveal` + компонент `ScrollReveal`) с одним общим `IntersectionObserver` на страницу; framer-motion и другие библиотеки не подключаются.
2. Состояние анимации передаётся через `data-reveal` / `data-reveal-visible` атрибуты, а CSS-правила объявлены через `:global(...)` внутри CSS-модуля — чистые attribute-селекторы без локального класса Turbopack отклоняет как «не чистые».
3. В мобильных медиазапросах правило `[data-reveal-visible="true"]` повторяется после переопределений начального `transform` — при равной специфичности решает порядок источника, иначе элементы «застревают» в начальном состоянии.
4. Хук возвращает callback ref и пропсы для spread (`{...reveal}`), а не RefObject: правило `react-hooks/refs` запрещает доступ к `.current` во время рендера.
5. `.page` демо-страницы получил `overflow-x: clip`, потому что hero-коллаж с `transform: scale(1.2)` создавал горизонтальный overflow на мобильном.

Почему:

ТЗ `docs/plankimi.md` запрещает новые тяжёлые библиотеки ради reveal-анимаций и требует анимировать только `opacity`/`transform` с поддержкой `prefers-reduced-motion`. Собственный модуль воспроизводит существующий паттерн `HomeMotion` и устраняет класс ошибок, найденных при верификации (застревание transform на мобильном, overflow).

Что ещё нужно доработать:

1. Ручная визуальная проверка на реальных устройствах (iOS Safari, Android Chrome).
2. Замер Core Web Vitals после выкладки, чтобы подтвердить отсутствие регресса.

## 2026-07-06

### Решение: мобильная раскладка финальной открытки использует container queries, дебаг-панель — body-атрибуты

Принято:

1. Для mobile-стилей финальной открытки (`/gift/[finalSlug]`) использовать CSS Container Queries по ширине `.shell`, а не `@media (max-width: 899px)` по viewport. Это позволяет видеть мобильную раскладку в дебаг-режиме на широком экране.
2. Дебаг-панель scrapbook-шаблона управляет состоянием через `data-scrapbook-debug-active` и `data-scrapbook-mobile-preview` на `<body>`, чтобы позиционировать панель относительно карточки, а не перекрывать её.
3. При включённом `debugAssets=1` mobile preview активен по умолчанию; панель можно свернуть/развернуть.
4. `.shell` в mobile preview ограничен `max-width: 430px` и переопределяет десктопное правило `.assetDebugActive .shell`, чтобы карточка не разъезжалась за пределы мобильного окна.
5. Исправлен баг с блокировкой прокрутки в `GiftIntro`: `overflow: hidden` сбрасывается при `alreadyOpened` и при размонтировании компонента.

Почему:

Дебаг-панель и mobile preview нужны для верификации мобильной вёрстки на ПК. Без container queries карточка оставалась desktop-ширины, а без переопределения ширины `.shell` активный debug-режим раздвигал карточку за пределы мобильного окна.

Что ещё нужно доработать:

1. Дебаг-панель: улучшить UX управления видимыми элементами, импортом/экспортом конфигурации и подсветкой выбранного ассета.
2. Проверить container queries во всех браузерах целевой аудитории (поддержка есть в современных Chrome/Safari/Firefox, но нужно убедиться, что нет fallback-проблем).
3. Дополнить ручное тестирование: открыть `/gift/[finalSlug]?debugAssets=1` с галочкой Mobile preview, без неё, и на реальном мобильном устройстве.

### Решение: аналитика beta остаётся собственной и безопасной для персональных данных

Принято:

1. Хранить только события пользовательского пути из разрешённого списка и технические идентификаторы.
2. Не сохранять в телеметрии тексты поздравлений, имена, email, токены управления, одноразовые ссылки и сырые ответы AI.
3. Использовать PostgreSQL в production и игнорируемый JSON-файл локально, не подключая внешнего поставщика аналитики до проверки beta.
4. Показывать операционную сводку только роли `admin` на `/admin/analytics`.
5. Сбой хранилища телеметрии не должен блокировать создание открытки, отправку поздравления или публикацию.
6. Не включать оплату в этот этап.

Почему:

Для бесплатной beta достаточно видеть потери воронки и критические сбои, не создавая вторую систему персональных данных и не расширяя инфраструктуру до подтверждения продуктового сценария.

### Решение: напоминания используют два служебных письма и сценарий зависит от близости события

Принято:

1. Сразу после сохранения отправлять подтверждение; для событий через 8+ дней напоминать за 7 дней, через 3–7 дней — следующим утром, через 1–2 дня отдельное напоминание не отправлять.
2. Для близких событий усиливать CTA создания открытки сейчас.
3. Отменять запланированное письмо только после явного подтверждения на странице, чтобы почтовые сканеры ссылок не отменяли его автоматически.
4. Формировать стабильный токен отмены через отдельный `REMINDER_TOKEN_SECRET`, сохраняя в БД только хеш.
5. Ограничить доставку пятью попытками, восстанавливать зависшие записи и использовать стабильные idempotency keys Resend.

Почему:

Одинаковое напоминание не подходит событиям разной срочности, а внешняя доставка требует защиты одновременно от дублей, аварийного перезапуска и автоматического открытия ссылок почтовыми системами.

## 2026-06-23

### Решение: основной старт MVP создает пустой черновик сразу с лендинга

Принято:

1. Кнопка `Создать открытку` на `/` должна создавать новый пустой черновик.
2. После создания пользователь сразу попадает в `/manage/[manageToken]`.
3. Старый `/create` можно оставить как резервный/исторический экран, но он больше не является главным стартовым путем.

Почему:

Пользователь ожидал, что стартовый CTA открывает новый рабочий черновик, а не старую форму создания. Это соответствует текущему MVP-flow: сначала быстро создать карточку, потом заполнять основу и оформление в управлении.

### Решение: следующий продуктовый фокус — экран участника

Принято:

1. Первичный дизайн финальной открытки считается зафиксированным.
2. Следующая работа — улучшение `/join/[slug]`, где участник пишет поздравление.
3. Skill `make-interfaces-feel-better` установлен локально, но если Codex не видит его как активный skill, можно читать его `SKILL.md` вручную и применять принципы.

Почему:

Конверсия участника в отправку поздравления важнее дальнейшей визуальной шлифовки открытки на этом этапе MVP.

## 2026-06-13

### Решение: зафиксировать постоянные правила проекта в репозитории

Принято:

1. Все значимые решения фиксируются в файлах проекта.
2. Обязательны тесты для прикладного кода.
3. Обязательно логирование ключевых сценариев, AI, платежей и ошибок.
4. Нужно регулярно делать локальные git-коммиты.

Почему:

Чтобы не терять договоренности и не пересобирать контекст заново в каждой сессии.

### Решение: сузить продуктовый фокус MVP 1.0

Принято:

1. Основные сценарии первого релиза — `учитель`, `воспитатель`, `коллега`.
2. На старте использовать один тариф.
3. Делать акцент на финальном эмоциональном результате, а не на “просто ссылке”.

Почему:

Это уменьшает распыление и повышает шанс быстрее проверить платежеспособный спрос.

### Решение: использовать 4 базовых шаблона вместо большого каталога

Принято:

1. `Теплый Классический`
2. `Командный Современный`
3. `Праздничный Яркий`
4. `Нежный Личный`

Почему:

Рынок подтверждает, что ранние версии таких продуктов могут успешно работать на ограниченном числе базовых стилей, если сильны контент и финальный опыт вручения.

### Решение: закрепить инженерную дисциплину на уровне проекта

Принято:

1. Тесты обязательны для прикладного кода.
2. Логирование считается обязательной частью реализации, а не дополнительной опцией.
3. Рабочие договоренности хранятся в репозитории.

Почему:

Это снижает риск забытых требований и делает дальнейшую разработку предсказуемой.

### Решение: взять Next.js как основу MVP-каркаса

Принято:

1. Основной стек каркаса — `Next.js + TypeScript`.
2. Для тестов использовать `Vitest`.
3. Базовое структурированное логирование реализовать своими средствами без внешней тяжелой зависимости.

Почему:

Это быстрый и достаточно гибкий стек для публичного лендинга, продуктовых экранов, серверной логики и дальнейших интеграций.

### Решение: на этапе MVP хранить черновики локально в файловом хранилище проекта

Принято:

1. Для текущего этапа черновики открыток сохраняются в локальный `data/cards.json`.
2. Файл с пользовательскими черновиками не коммитится в git.
3. Репозиторий чтения и записи отделен от UI и API-слоя.

Почему:

Это самый быстрый способ получить рабочий сценарий создания открытки до подключения полноценной базы данных.

### Решение: хранить поздравления локально тем же способом до подключения базы данных

Принято:

1. Поздравления на текущем этапе сохраняются в локальный `data/contributions.json`.
2. Слой хранения поздравлений отделен от API и UI.
3. На странице участника пока показываются только видимые сохраненные поздравления.

Почему:

Это позволяет быстрее проверить живой сценарий сбора поздравлений до внедрения полноценной БД и модерации.

### Решение: ужесточить минимальное качество поздравлений

Принято:

1. Простого порога по символам недостаточно.
2. Поздравление должно быть не короче 20 символов.
3. Поздравление должно содержать хотя бы 3 слова.
4. Слишком общие тексты вроде `Поздравляю!` должны отклоняться.

Почему:

Формально короткие фразы проходят валидацию, но не дают той эмоциональной ценности, за которую пользователь потом будет готов платить.

### Решение: сначала внедрить внутренний AI-слой, а не ждать внешнюю модель

Принято:

1. AI-помощник на текущем этапе работает как внутренний генератор черновиков.
2. Слой AI отделен через собственный API, валидацию, лимиты и логирование.
3. Позже этот слой можно заменить на внешнюю модель без полной переделки интерфейса участника.

Почему:

Так мы быстрее проверяем UX и пользу AI-помощника, не блокируясь на внешней интеграции.

### Решение: дать организатору управление поздравлениями до оплаты

Принято:

1. У организатора есть секретная ссылка управления.
2. До этапа оплаты он уже может скрывать, возвращать и удалять поздравления.
3. Предпросмотр и напоминание для чата доступны до внедрения отдельной платежной логики.

Почему:

Именно этот слой делает продукт полезным уже до монетизации и позволяет проверить, чувствует ли организатор реальный контроль над результатом.

### Решение: считать грубую подстановку в AI-тексте дефектом качества

Принято:

1. AI-черновик не должен механически вставлять ответы пользователя в текст.
2. Роль участника должна влиять на формулировки.
3. Сомнительные личные детали должны отбрасываться, а не переноситься в поздравление автоматически.
4. Качества должны встраиваться естественно, чаще через существительные и более живые конструкции.

Почему:

Если AI выглядит как конструктор шаблонов с грубыми вставками, пользователь не воспринимает это как настоящую помощь и теряет доверие к продукту.

### Решение: считать роль участника и вариативность частью качества AI

Принято:

1. Роль участника должна менять тон и открывающие формулировки.
2. Если пользователь по ошибке вводит роль получателя вместо своей, AI не должен слепо тащить это в текст.
3. Повторные генерации по одной открытке должны заметно отличаться по формулировкам.

Почему:

Без этого AI быстро начинает выглядеть механическим и однообразным даже при рабочей технической реализации.

### Решение: отказаться от слишком узких “профессиональных” шаблонов как главной логики

Принято:

1. Шаблоны должны быть в первую очередь универсальными визуальными системами.
2. Кого именно поздравляют, определяется контентом, текстом, медиа и композиционными блоками, а не названием шаблона.
3. Примеры и демо-открытки могут быть тематическими, но базовые шаблоны не должны жестко привязываться к профессии или одному сценарию.

Почему:

Пользователь ценит не “шаблон для воспитателя” сам по себе, а сильный визуальный результат, который адаптируется под конкретного человека.

### Решение: строить путь создания вокруг живого повода, а не вокруг справочника ролей получателя

Принято:

1. На форме создания организатор сам пишет повод или контекст поздравления своими словами.
2. Выбор `Личная`, `Командная`, `Праздничная` трактуется как формат и настроение будущей открытки, а не как профессия получателя.
3. В участническом и финальном экранах основной контекст берется из `occasionText`, а не из старых категорий `учитель / воспитатель / коллега`.
4. Старые значения в данных сохраняются только ради обратной совместимости с уже созданными черновиками.

Почему:

Так сценарий становится понятнее, универсальнее и ближе к реальной ценности продукта: люди собирают открытку для конкретного человека по конкретному поводу, а не подбирают “подходящую профессию” из списка.

### Решение: упростить AI-помощник до одного поля мыслей + выбора стиля

Принято:

1. Участник не выбирает роль, качества и пожелания по отдельности через селекты и чекбоксы.
2. Вместо этого он пишет мысли в свободной форме в одно поле.
3. AI-помощник преобразует этот сырой текст в 3 более аккуратных варианта поздравления.
4. Выбор блоков финального экрана остается задачей организатора и будет жить на странице управления, а не в форме участника.

Почему:

Так путь становится естественнее: пользователь думает про человека и свои слова, а не про заполнение искусственного конструктора.

### Решение: редактирование поздравлений сначала отдать организатору

Принято:

1. Первый рабочий слой редактирования появляется в админке организатора.
2. Организатор может править текст поздравления перед финальной открыткой без изменения автора и остальных метаданных.
3. Редактирование участником собственного текста можно добавить позже отдельным шагом, если это подтвердится как реальная потребность.

Почему:

Сейчас важнее дать контроль над качеством итоговой открытки тому, кто собирает результат и отвечает за финальный вид.

### Решение: выбор блоков финального экрана хранить в открытке и настраивать в админке

Принято:

1. Организатор управляет необязательными блоками финального экрана на своей секретной странице.
2. Настройки сохраняются в данных самой открытки.
3. Обязательные блоки `hero`, `messages`, `closing` нельзя выключить.
4. Необязательные блоки можно включать и выключать только если для них реально есть контент.

Почему:

Так управление итоговым экраном становится понятным и устойчивым: выбор не теряется, сразу влияет на `/gift/...` и не дает собрать пустые секции.

### Решение: усиливать финальный экран через выразительный визуальный ритм, а не через новые сущности

Принято:

1. Полировать финальный экран в первую очередь стилем, композицией и иерархией блоков.
2. Сохранять текущую block-based архитектуру без усложнения модели данных ради визуального эффекта.
3. Делать главный вау-эффект через hero, карточки поздравлений, цитаты и финальный аккорд.

Почему:

На этом этапе продукту важнее выглядеть как цельный цифровой подарок, чем наращивать новые типы данных и сложные редакторы.

## 2026-06-15

### Decision: hidden and active contributions are separate working groups

Accepted:

1. The organizer should always see whether a contribution is active or hidden directly on the card.
2. Hidden contributions should not stay mixed into the active working list.
3. When a contribution becomes hidden, it moves to the end of the list.
4. When a hidden contribution becomes active again, it moves to the end of the active group, before hidden contributions.
5. The movement is intentional product behavior, not just a visual sort: the saved order must match the visible grouping after refresh.

Why:

The organizer is moderating a final card, not just toggling database flags. Active messages are the working set; hidden messages are retained but visually de-prioritized.

### Decision: drag-and-drop must feel handle-based

Accepted:

1. Dragging contributions and composition blocks should feel attached to the drag handle, not to the center of the card.
2. Fixed blocks in card composition remain non-draggable.
3. Drag feedback should communicate the future insertion point before drop.

Why:

The editor should feel like a premium card constructor. If the dragged item jumps to the center, the interaction feels technical and imprecise.

### Decision: status changes collapse the edited contribution card

Accepted:

1. A contribution card collapses before it moves because of a visibility-status change.
2. This avoids moving a large expanded editor through the list.

Why:

The status action is moderation, not text editing. Once the organizer changes visibility, the card should return to compact list mode and move cleanly to its new group.

Why:

The project now needs proof that the greeting-card flow works end to end. Visual quality matters, but more polishing has lower leverage than making the prototype usable by a real organizer.

### Decision: use lifecycle statuses before accounts

Accepted:

Why:

Statuses give enough operational control for MVP without introducing accounts, roles, or a service admin panel too early.
