# MVP-flow после первичного дизайна

Дата: 2026-06-23

Актуальный handoff для нового чата: `docs/PROJECT_HANDOFF_2026-06-23.md`.

## Продуктовый фокус

Первичный визуальный дизайн scrapbook-открытки зафиксирован. Следующий этап — не дальнейшая полировка, а рабочий жизненный цикл одной открытки:

1. организатор создает открытку;
2. получает ссылку управления;
3. отправляет участникам ссылку сбора поздравлений;
4. участники добавляют поздравления;
5. организатор модерирует, выбирает главное поздравление и проверяет финальную версию;
6. получатель открывает финальную ссылку.

## URL-структура MVP

Используем один основной домен без поддоменов:

1. `/` — лендинг.
2. `/create` — создание открытки.
3. `/join/[slug]` — страница для участников, которые пишут поздравление.
4. `/manage/[manageToken]` — управление открыткой для организатора.
5. `/preview/[manageToken]` — предпросмотр для организатора.
6. `/gift/[slug]` — финальная открытка для получателя.

Production base URL задается через `NEXT_PUBLIC_SITE_URL=https://darislova.ru`.
Ссылки нужно строить через `src/lib/routes/card-links.ts`, а не хардкодить домен или `localhost` в компонентах.

## Что уже есть

1. Стартовый экран и создание открытки.
2. Экран управления и модерации.
3. Предпросмотр.
4. Финальная открытка.
5. Страница участника.
6. Scrapbook-шаблон с настроенным визуальным конфигом.
7. Переключаемое хранилище: JSON локально без `DATABASE_URL`, PostgreSQL при заданном `DATABASE_URL`.
8. Production Docker Compose заготовка для отдельного VPS stack рядом с Prognozist.

## Что добавлено в MVP-flow

1. Статусы открытки:
   - `draft` — черновик;
   - `collecting` — сбор поздравлений;
   - `ready` — готова к отправке;
   - `closed` — сбор закрыт.
2. Панель ссылок на экране организатора:
   - участникам;
   - организатору;
   - получателю.
3. Копирование ссылок из панели организатора.
4. Закрытие сбора поздравлений через статус `closed`.
5. Базовые действия в финале:
   - реакция “Спасибо”;
   - сохранение через печать/системное сохранение страницы;
   - переход к созданию новой открытки.
6. Ограничение медиа: до 7 фото на одну открытку.
7. Кнопка “Создать открытку” создает новый пустой черновик; экран управления больше не подставляет демо-имена в пустые поля.
8. Локальный storage-слой для фото: файлы сохраняются в `public/uploads/cards`, метаданные — в JSON или PostgreSQL в зависимости от режима.
9. Link-helper для MVP URL: `getGiftUrl`, `getJoinUrl`, `getManageUrl`, `getPreviewUrl`.
10. Production compose: `capsule-web` + отдельный `capsule-postgres`, uploads через bind mount.
11. Основная кнопка `Создать открытку` на лендинге создает новый пустой черновик через server action и сразу открывает `/manage/[manageToken]`.
12. Старый `/create` остается в проекте, но не является главным стартовым путем.

## Решение по фото

На ближайший MVP в одной открытке может быть максимум 7 фото:

1. до 3 фото рядом с блоком поздравлений;
2. до 3 фото в блоке “Моменты”.

Вертикальный вариант блока поздравлений использует один слот вместо горизонтальной группы, но общий лимит библиотеки всё равно остается 7 фото. Это упрощает хранение, ускоряет загрузку открытки и делает объём данных на VPS предсказуемым.
