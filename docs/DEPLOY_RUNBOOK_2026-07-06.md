# Runbook накопленного production-релиза

Дата: 2026-07-06.

Этот runbook выполняется только после закрытия локального этапа A из `docs/plan.md`. Значения секретов в команды, логи и Git не копируются.

## 1. До изменения VPS

1. Зафиксировать текущий production commit и состояние контейнеров.
2. Проверить свободное место на диске.
3. Запустить nightly backup PostgreSQL и uploads.
4. Проверить checksum и выполнить `infra/scripts/verify-backup-restore.sh`.
5. Убедиться, что сохранён предыдущий рабочий image/commit.
6. Проверить наличие всех переменных из `.env.production.example`, включая admin, reminders, email и AI.

## 2. Релиз

1. Получить подготовленный release commit.
2. Проверить `docker compose ... config`.
3. Собрать новый web image, не удаляя предыдущий.
4. Применить миграции `0010`–`0015`; повторный запуск должен быть безопасен.
5. Перезапустить web и проверить health обоих контейнеров.
6. Убедиться, что Caddy подключён к постоянной общей сети.
7. Проверить HTTPS для `darislova.ru` и `www.darislova.ru`.

## 3. Настройки после запуска

1. Проверить production admin credentials и при необходимости ротировать их.
2. Настроить `CRON_SECRET`, `REMINDER_TOKEN_SECRET` и reminder cron на `04:00 UTC`.
3. Проверить nightly backup cron и пути логов.
4. Добавить ежедневный вызов `infra/scripts/run-card-retention.sh` после backup и проверить его лог.
5. Не включать `PUBLICATION_MODE=paid` до отдельного платёжного релиза.

## 4. Smoke-check

1. Создать новую открытку.
2. Проверить `/manage`, `/join`, AI, поздравление и фото.
3. Опубликовать и открыть `/gift` на desktop и телефоне.
4. Проверить organizer magic link и повторную отправку.
5. Войти в `/admin`, проверить роли, аналитику, ошибки и обращения.
6. Проверить reminder confirmation, cancellation и ручной безопасный вызов cron.
7. Перезапустить web и убедиться, что PostgreSQL и uploads сохранились.
8. Сделать новый backup и снова проверить восстановление.

## 5. Rollback

Rollback нужен при недоступности сайта, ошибке миграции, потере данных, невозможности создать/открыть открытку или массовом сбое email/AI.

1. Остановить новый web-контейнер, не удаляя volumes и uploads.
2. Вернуть предыдущий рабочий commit/image и прежний env.
3. Если миграция только добавляла совместимые таблицы/поля, базу назад не откатывать автоматически.
4. Восстанавливать PostgreSQL только при подтверждённом повреждении данных и только из проверенного backup.
5. После восстановления повторить health-check и минимальный пользовательский сценарий.
6. Зафиксировать причину, использованный backup и итог в delivery log.

## 6. Условие перехода к оплате

Бесплатная beta должна дважды пройти полный production-flow, пережить перезапуск, один цикл cron/backup и тестовое восстановление. После этого начинается отдельный платёжный этап C.
