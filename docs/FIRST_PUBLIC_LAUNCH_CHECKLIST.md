# Чек-лист первого публичного запуска Slovesto

Это последовательность для первого публичного запуска на `slovesto.ru`. Старый домен и его данные не участвуют в запуске; это не миграция. Выполняйте пункты по порядку и отмечайте результат в release-log.

Смежные подробные инструкции: `docs/REBRANDING_AND_DOMAIN_MIGRATION.md`, `docs/DEPLOY_RUNBOOK_2026-07-06.md` и `docs/VPS_DEPLOYMENT_NOTES.md`.

## 1. Локальная готовность

- [ ] Проверить состав релиза и исключить секреты, `.env.local`, production uploads и данные из коммита.
- [ ] Запустить `npm run lint`, `npm test` и `npm run build`.
- [ ] Запустить `npm run brand:check` и визуально принять header на desktop и mobile.
- [ ] Зафиксировать проверенный набор изменений отдельным release commit.

**Стоп-условие:** любая ошибка проверок или неутверждённый интерфейс. Исправить локально и повторить этот раздел.

## 2. Домен и почта

- [ ] Направить `slovesto.ru` на production-сервер.
- [ ] Настроить `www.slovesto.ru` с постоянным 301 на `https://slovesto.ru` с сохранением path и query.
- [ ] Выпустить и проверить TLS-сертификаты для обоих доменов.
- [ ] В production env задать `NEXT_PUBLIC_SITE_URL=https://slovesto.ru` и `EMAIL_FROM="Slovesto <hello@mail.slovesto.ru>"`.
- [ ] Не задавать `EMAIL_REPLY_TO`, пока для адреса не настроен принимающий ящик или пересылка.
- [ ] Убедиться, что `mail.slovesto.ru` подтверждён в Resend и письма отправляются с нужного имени.

**Контроль:** `https://slovesto.ru` открывается без предупреждений; вариант с `www` возвращает 301 на основной домен.

## 3. Резервная копия и тестовые данные

- [ ] Зафиксировать текущий commit, образы и состояние контейнеров на VPS.
- [ ] Создать backup PostgreSQL и `public/uploads`; сохранить путь, время и checksum.
- [ ] Проверить восстановление backup по `infra/scripts/verify-backup-restore.sh`. Если backup снят до ещё не применённых миграций релиза, передайте текущее количество миграций сервера через `MIN_MIGRATIONS`.
- [ ] Запустить только отчёт об очистке: `bash infra/scripts/clear-test-data-for-public-launch.sh`.
- [ ] Сверить количества тестовых сущностей и файлов с ожидаемыми.

После подтверждения backup и отчёта владелец запуска может отдельно выполнить необратимую очистку:

```bash
BACKUP_CONFIRMED=YES CONFIRM_TEST_DATA_CLEANUP=DELETE_TEST_DATA \
  bash infra/scripts/clear-test-data-for-public-launch.sh
```

Скрипт очищает пользовательские тестовые данные и карточные uploads, но сохраняет migrations, шаблоны, администраторов и секреты. Не выполняйте эту команду без проверенного backup.

## 4. Выкладка

- [ ] На VPS получить release commit.
- [ ] Проверить production-конфигурацию Docker Compose без вывода секретов.
- [ ] Собрать новый web image, сохранив предыдущий рабочий image для rollback.
- [ ] Применить миграции и перезапустить сервисы по `docs/DEPLOY_RUNBOOK_2026-07-06.md`.
- [ ] Проверить health-check, логи приложения и доступность PostgreSQL/uploads.

**Стоп-условие:** недоступность, ошибка миграции, отсутствие uploads или массовые ошибки email/AI. Сразу перейти к rollback из runbook; базу не откатывать автоматически для совместимых миграций.

## 5. Публичная проверка

- [ ] Открыть главную, пример, `robots.txt`, `sitemap.xml` и проверить canonical `https://slovesto.ru`.
- [ ] Проверить favicon, логотип, OG preview и ссылку из письма.
- [ ] Создать тестовую открытку: текст, фото, AI-помощь, публикация и просмотр `/gift` на desktop и телефоне.
- [ ] Проверить `/manage`, `/join`, magic link организатора и повторную отправку письма.
- [ ] Проверить admin, поддержку, reminder confirmation/cancellation и безопасный ручной вызов cron.
- [ ] Перезапустить web и подтвердить сохранность базы и uploads.
- [ ] Создать новый backup после запуска и проверить восстановление.

## 6. После открытия доступа

- [ ] Добавить `slovesto.ru` в Google Search Console, Яндекс Вебмастер и Яндекс Метрику.
- [ ] Отправить `https://slovesto.ru/sitemap.xml` в панели вебмастеров.
- [ ] В течение первых суток отслеживать ошибки обхода, health, delivery email, cron и логи приложения.
- [ ] Не включать `PUBLICATION_MODE=paid` до отдельного платёжного релиза и двух успешных полных beta-циклов.

## Результат запуска

Запуск считается завершённым, когда основной домен и редирект `www` работают по HTTPS, тестовые данные очищены только после проверенного backup, пройден пользовательский сценарий, а новый backup успешно восстановлен.
